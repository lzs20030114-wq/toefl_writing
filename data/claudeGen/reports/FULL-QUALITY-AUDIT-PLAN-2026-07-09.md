# 全库题目质量监测 — 最小成本执行计划（2026-07-09，仅计划未执行）

**对象**：全库 13 型 2855 题（写作 1059 / 阅读 893 / 听力 796 / 口语 107，07-09 main 实测）。
**目标**：一次体检查五类病——结构坏题、答案错键、拟真度漂移、重复、音频问题——并留下
**增量审计台账**，让以后的「全库监测」只需审新增，成本从 O(全库) 降为 O(增量)。

**设计总原则：token 花在只有 LLM 能干的事上。** 五类病里三类（结构/重复/音频）确定性代码
就能全查，一类（答案错键）用便宜一个数量级的 DeepSeek，只有拟真度需要高质量 LLM——而拟真度
是**批分布属性不是逐题属性**，抽样即可，不必全量。

---

## 四层漏斗

### L0 — 确定性全扫（0 token，全库 100% 覆盖，~分钟级）

| 检查 | 工具（现成度） | 覆盖 |
|---|---|---|
| 逐题结构校验 | 各型 validator（100% 现成，13/13 型已有映射） | 全库 |
| 内容重复/近重复 | contentDedup 指纹（dedup-banks.mjs 口径，现成） | 全库 |
| 批级形态统计：答案位偏移、正确项最长率 | 新写 ~100 行统计器（审查报告 §4 口径已定义） | MCQ 全库 |
| 难度字段覆盖、字数带 vs realExam profile | scoreBatch 量尺（现成）+ eval-profiles | 全库 |
| 音频完整性：audio_url 覆盖率、指向的 storage 对象存在性 | 上次孤儿清理的 id 反查口径复用 | 听力/口语 |
| gate harness 冻结带 | scripts/cli/enforce-gates.mjs REPORT 模式（现成） | 已注册维度 |

产出：结构坏题清单 + 全库统计报告。**新代码量：一个 runner 脚本把现成件串起来。**

### L1 — 答案正确性二审（DeepSeek API，≈¥15-40 一次性，0 Claude token）

- 客观题**全量**独立答题比对（fail-closed 出嫌疑清单，不自动删）：
  - 阅读 ap/rdl/ctw：`answerAuditor` **现成**（merge 层同款，直接对存量跑）；
  - 听力 lcr/lc/la/lat：auditor 存在但从未接线（§7 P1-9）——本次顺手接上，一活两干；
  - BS：答案唯一性/可解性走现有 consistency 检查口径。
- 量级：~2400 客观题 × ~1k tokens ≈ DeepSeek ¥15-40（按现价粗算，执行前用 /cost 精算）。
- 错峰限速（45/min 限流内），跑一晚完事。

### L2 — 拟真度抽样评审（Claude 夜间 routine，订阅边际成本≈0，分 2-3 晚摊销）

- **抽样而非全量**：每库分层抽 ~30 题（按批次 run-id 时间分层，新批次加权），
  13 库 ≈ 390 题。统计功效：单库 30 样本足以检出 >15pp 的分布级漂移——拟真度问题
  从来是批属性（某晚某 prompt 退化），不是单题彩票。
- 评审维度直接用 `docs/eval-spec/*.md` 各型维度表出结构化 verdict（像/不像/病因），
  对照锚 = `data/realExam2026/`。
- 主观题（disc/email/interview/repeat）没有客观答案，全靠这层 + L0 统计。
- **跑在夜间 routine 时段**（今天白天撞过 session limit 的教训）：拆成每晚 4-5 库，
  挂在 R1 完成之后执行，2-3 晚跑完，不占白天交互额度。

### L3 — 人工复核（你，~30 分钟）

- L0-L2 汇总 top 嫌疑题（每库 ≤10，按严重度排序）渲染成一页；
- 你只看嫌疑犯，逐条拍板：删 / 修 / 留。处置走现有工具（dedup-banks 同款删除脚本 + 台账）。

## 成本汇总

| 资源 | 用量 | 说明 |
|---|---|---|
| Claude 交互 token（白天会话） | ~1 次中等会话 | 只做：L0 runner 编排 + L1 接线 + 报告汇总。L0/L1 本身是脚本，跑起来不吃 token |
| Claude routine（订阅） | 2-3 晚各 +20 分钟 | L2 抽样评审，边际 ¥0，避开 session limit |
| DeepSeek API | ¥15-40 一次性 | L1 全量答案二审（执行前 /cost 精算） |
| 其他（TTS/存储） | ¥0 | 只读不生成 |
| **以后每次复检** | **≈增量部分** | audited-ledger.json（题 id+内容指纹+审计版本）建好后，只审新题 |

## 执行顺序与产出

1. L0 全扫（当天出报告）→ 依结果收窄 L1/L2 范围；
2. L1 答案二审（一晚）+ 听力 auditor 接线（顺手清 P1-9）；
3. L2 抽样评审排 routine（2-3 晚）；
4. L3 你复核拍板 → 处置 → 台账落盘。
- 产出目录：`data/claudeGen/reports/FULL-AUDIT-<date>/`（报告 + 嫌疑清单 + audited-ledger.json）。
- 日历时间 ~3 天（夜间摊销），白天占用你 ~30 分钟。

## 风险与取舍（明说）

- 听力 auditor 接口与阅读不同（审查报告 §5.1），L1 接线有半天工作量，但这是 P1-9 本来就欠的账；
- 拟真度抽样会漏「单题级」偶发怪题——但那类题 L0 结构检查和 L1 答案审计能兜住大半，
  剩余靠错题本/用户反馈回流，全量 LLM 评审 2855 题的成本（≈7-10 倍 token）买不来对应收益；
- 回忆卷锚薄的题型（RDL 零锚、interview 锚弱）拟真度结论会标「低置信」，不下硬结论。
